## 1. 项目概览

这是一套 **ROS2 工作空间**，用于控制一台 **Lynx（猞猁）履带式差速越野车**——通过独立控制左右两条履带的驱动轮转速来实现前进与转向（原理类似坦克）。

当前工程实现的核心功能是：**沿一条内部生成的参考轨迹自动行驶**，并带有在线学习与扰动补偿能力。工程中**尚无视觉感知模块，也无履带升降的命令通道**（只有履带角度的只读反馈）——这正是 VLA 需要补充的部分。

代码分为三个 ROS2 包：

| 包 | 语言 | 构建系统 | 角色 |
|---|---|---|---|
| `cav_msgs` | ROS IDL | ament_cmake | **数据契约**：定义节点间通信的消息格式 |
| `cav_bywire_lvtan` | C++17 | ament_cmake | **底层驱动**：ROS 消息 ↔ CAN 总线的双向翻译 |
| `learning_preview_controller` | Python | ament_python | **上层控制器**：路径跟踪算法（Preview-LQR / MPC）|

> 目录中的 `src/无解.zip`、`learning_preview_controller.zip` 仅为源码快照备份，与当前源码一致，可忽略。

## 2. 系统架构与数据流

系统运行时共有 **4 个 ROS 进程**：`ros2_socketcan` 的 `to_can` 与 `from_can`（CAN 总线桥）、C++ 驱动节点 `lvtan_bywire_node`、Python 控制器节点 `learning_preview_controller`。

```
  RTK/GNSS ──/car/gps (Gpybm)───────────────┐
                                            ▼
  车辆 CAN 反馈                       ┌──────────────────────────┐
  0x4D1~0x4D4 ──/from_can_bus──────► │  C++ 驱动节点             │
                                     │  lvtan_bywire_node       │─/vehicle/vehicle_state─┐
                                     │  (cav_bywire_lvtan)      │   (VehicleState 全状态) │
                       ┌────────────►│                          │                        ▼
                       │             └──────────────────────────┘          ┌──────────────────────────────┐
                       │                        ▲                          │  Python 控制器节点            │
                       │                        │ /vehicle/control2bywire  │  learning_preview_controller  │
  车辆 ◄─/to_can_bus───┘                        └────(Control 控制指令)─────│  (50Hz 控制环)                │
        0x4C1~0x4CF                                                        └──────────────────────────────┘
```

**闭环节奏**：

- C++ 驱动侧：一个 **50Hz 定时器** (`FREQ=50`) 持续把最新指令编码成 CAN 帧发往车辆；RTK 回调到来时融合最新 CAN 反馈，组装并发布 `VehicleState`。
- Python 控制器侧：一个 **50Hz 定时器**（`control_period=0.05s`）读取最新 `VehicleState`，计算左右轮速指令，发布 `Control`。
- 两侧都以 20ms 为周期，但各自独立触发（非严格同步）。

**关键设计点**：控制器的唯一执行输出是 **左右驱动轮角速度 `W_l, W_r`（rad/s）**。所有转向、直行都由这两个量的差速/共模实现。

## 3. 话题与 CAN 帧总表

### ROS 话题

| 话题 | 消息类型 | 发布者 | 订阅者 | 说明 |
|---|---|---|---|---|
| `/car/gps` | `Gpybm` | RTK 驱动(外部) | C++ 驱动 | RTK 原始定位 |
| `/from_can_bus` | `can_msgs/Frame` | socketcan | C++ 驱动 | 车辆 CAN 反馈 |
| `/to_can_bus` | `can_msgs/Frame` | C++ 驱动 | socketcan | 发往车辆的 CAN 指令 |
| `/vehicle/vehicle_state` | `VehicleState` | C++ 驱动 | Python 控制器 | 车辆完整状态 |
| `/vehicle/control2bywire` | `Control` | Python 控制器 | C++ 驱动 | 控制指令 |
| `/cav_path_plan/result` | `PlanedPath` | (外部/未用) | C++ 驱动 | 旧路径接口，当前控制器不发 |

### CAN 帧协议（由代码逆推）

**下行（PC → 车，0x4Cx）**，在 [can_input.cpp](../src/cav_bywire_lvtan/src/can_input.cpp) 中编码：

| CAN ID | 内容 | 字节布局 |
|---|---|---|
| `0x4C1` | RTK 经纬度 | B0-3 经度×1e7 (i32 LE)，B4-7 纬度×1e7 (i32 LE) |
| `0x4C2` | **核心控制帧** | B0 Stop(0-3)，B1 Go，B2-3 左轮 rpm×10 (i16 LE)，B4-5 右轮 rpm×10 (i16 LE)，B6=1，B7=0 |
| `0x4C3` | 本地位姿 | B0-2 x，B3-5 y (u24 LE, ×100+8e6)，B6-7 yaw (×1000+5000) |
| `0x4C4~0x4CD` | 路径点(10 个) | 每帧 x/y/speed |
| `0x4CE/0x4CF/0x4C0` | 路径曲率半径 | i16 LE ×100 |

**上行（车 → PC，0x4Dx）**，在 [can_output.cpp](../src/cav_bywire_lvtan/src/can_output.cpp) 中解码：

| CAN ID | 内容 | 解码 |
|---|---|---|
| `0x4D1` | 速度/轮速/电量 | B0-1 车速 `(u16-300)*0.01` m/s，B2-3 左轮 rpm `(u16-1200)*0.1`，B4-5 右轮 rpm，B6-7 SOC(%) |
| `0x4D2` | 状态/**履带角**/模式 | B0 drive_state(0差速/1刹车/2原地转向)，**B1 履带角 `data-90` 度**，B2 控制模式，B3-4 横摆角速度，B5-7 车身尺寸 |
| `0x4D3` | 故障码 | B0 整车，B1 左电机，B2 右电机 |
| `0x4D4` | 转向角 | B0-1 `(u16-1000)*0.1` 度 → rad |

> 🔑 **VLA 相关**：履带角 `track_angle`（范围校验在 [-50°, 80°]）**仅出现在上行 0x4D2 反馈中，没有任何下行帧能命令履带升降**。要让 VLA 输出真正作动，必须新增一条下行命令帧（需向车厂索取"写履带"CAN 协议）。

## 4. 包一：cav_msgs —— 数据契约

由 [CMakeLists.txt](../src/cav_msgs/CMakeLists.txt) 用 `rosidl_generate_interfaces` 把 5 个 `.msg` 编译成 C++/Python 双语言类型。

### 4.1 [VehicleState.msg](../src/cav_msgs/msg/VehicleState.msg)（134 行）——车辆完整状态

系统中最大的消息，C++ 驱动发布给控制器。字段分组：

- **任务信息**：`id / name / vehicle_state(0正常/1损坏/2低电) / type`
- **RTK 状态与定位**：`rtk_gps_longitude/latitude/altitude`、`rtk_gps_utm_x/y/z`（UTM 米）
- **速度**：`rtk_linear_enu_vx/vy/vz`（ENU 全局系）、`rtk_linear_vx/vy/vz`（车体系）、`speed_x/y/z`
- **关键位姿**：`x, y, z`、`heading`(yaw)、`pitch`、`roll`、`yaw_rate`
- **履带车专有反馈**：
  - `left_drive_wheel_speed / right_drive_wheel_speed`（左右驱动轮角速度，rad/s）
  - `left_drive_wheel_rpm / right_drive_wheel_rpm`
- **挡位** `gear_pos`（0-5 = NONE/PARK/REVERSE/NEUTRAL/DRIVE/LOW）
- **线控/CAN 反馈**：`throttle_* / brake_* / steer_*`、`by_wire_enabled`、`vehicle_mode`
- **电池/发动机**：`battery_volt / battery_soc / engine_speed`
- **故障码**：`vehicle_bug_code / vehicle_state_code`

> 🔑 **VLA 相关**：计算车轮打滑率所需的全部数据都在此消息内——`left/right_drive_wheel_speed`（轮速）与 `speed_x`（车体实速）。打滑率可用 `slip = (ω·r − v) / (ω·r)` 估算。

### 4.2 [Control.msg](../src/cav_msgs/msg/Control.msg)（30 行）——控制指令

Python 控制器发布给 C++ 驱动。关键字段：

- `left_drive_wheel_speed_cmd / right_drive_wheel_speed_cmd`：**左右驱动轮目标角速度（rad/s）——控制器真正下发的唯一执行量**
- `gear_cmd`：挡位（0-5）——注意：**当前 C++ 驱动接收但未使用此字段**
- `bywire_control_enable / emerg_brake / park_enable`：三个安全开关，C++ 据此决定 Stop/Go
- `throttle_cmd / brake_cmd / acceleration_cmd`：本履带车模式下未实际使用
- `ey_out / ephi_out`：横向误差、航向误差（仅用于记录/显示）

> 🔑 **VLA 相关**：此消息**没有履带升降字段**。VLA 输出可选择：(a) 给本消息加 `int16 track_gear_cmd` 字段；或 (b) 新建独立消息/话题（推荐，解耦度高）。

### 4.3 其余消息

- [Gpybm.msg](../src/cav_msgs/msg/Gpybm.msg)：RTK/GNSS 原始数据（经纬度、高度、航向 yaw、俯仰、各向速度、高斯坐标、基站坐标、状态位）。
- [PlanedPath.msg](../src/cav_msgs/msg/PlanedPath.msg) + [Point.msg](../src/cav_msgs/msg/Point.msg)：一串路径点（`t/x/y/v/heading/cr`）。**当前 Python 控制器不发布此话题**，路径由控制器内部生成；此接口是为 C++ 那套 0x4C4~0x4CF 旧协议保留的。

## 5. 包二：cav_bywire_lvtan —— C++ 底层驱动

唯一直接操作 CAN 总线的部分。核心是一个 ROS 节点 `lvtan_bywire_node`。构建配置见 [CMakeLists.txt](../src/cav_bywire_lvtan/CMakeLists.txt)：C++17，`file(GLOB SRC "src/*.cpp")` 编译 src 下全部 cpp 为单个可执行文件，并额外包含 `/opt/xproj_deploy/include_public`（部署环境的外部头文件）。

### 5.1 [main.cpp](../src/cav_bywire_lvtan/src/main.cpp)——入口

极简：初始化 rclcpp → 创建 `ROSNode` → 声明并读取 5 个参数（`veh_id / veh_type / veh_name / move_x / move_y`）→ `spin`。其中 `move_x / move_y` 是把 RTK 天线位置平移到车辆参考点的偏移量。

### 5.2 [ros_cav.h](../src/cav_bywire_lvtan/include/ros_cav.h) + [ros_cav.cpp](../src/cav_bywire_lvtan/src/ros_cav.cpp)——节点骨架与"接线"

`ros_cav.h` 声明了 `ROSNode` 类：两个数据结构体 `can_input`/`can_output`、全部订阅/发布句柄、以及各个编解码函数原型。

`ros_cav.cpp` 的构造函数是理解节点行为的钥匙——它建立了所有连接：

```cpp
pub_vs      = create_publisher<VehicleState>("/vehicle/vehicle_state", 2);
pub_can_cmd = create_publisher<can_msgs::Frame>("/to_can_bus", 2);

sub_control2bywire_msg = create_subscription<Control>("/vehicle/control2bywire", ...);   // 上层指令
sub_traj_plan_msg      = create_subscription<PlanedPath>("/cav_path_plan/result", ...);   // 旧路径
sub_can_feedback_msg   = create_subscription<Frame>("/from_can_bus", 20, ...);            // CAN 反馈
sub_rtk                = create_subscription<Gpybm>("/car/gps", 10, ...);                 // RTK

// 50Hz 定时器：这是 CAN 命令下发的心跳，也让"控制丢失计数"生效
can_command_timer = create_wall_timer(20ms, &ROSNode::publish_can_commands);
```

### 5.3 [dataModel.hpp](../src/cav_bywire_lvtan/include/dataModel.hpp)——车辆参数与核心结构体

- **频率**：`FREQ = 50`
- **车辆几何常量**：`LYNX_WHEEL_RADIUS = 0.292 m`、`LYNX_HALF_TRACK = 0.557 m`（半履带距）、`LYNX_L_WHEELS = 1.94 m`、`LYNX_STEER_RATIO = 17.0`
- **挡位宏** `GEAR_NONE/PARK/REVERSE/NEUTRAL/DRIVE/LOW = 0..5`
- **角度工具** `XM::Normalise_PI`（归一化到 [-π,π]）、`Normalise_2PI`
- **`CAN_INPUT_S`**（要发给车的）：左右轮速指令、经安全校验的 `*_can_cmd`、`Stop/Go`、RTK 经纬度、本地 x/y/yaw、路径点列表、以及 `loss_control_num`（控制丢失计数，被 50Hz 定时器累加、被控制回调清零）。
- **`CAN_OUTPUT_S`**（从车读到的）：车速、横摆角速度、车身尺寸、**`track_angle`（履带角，度）**、转向角、左右 rpm、SOC、`drive_state`、`contol_mode`（原文拼写保留）、故障码。

> 🔑 `track_angle` 与 `drive_state` 是与"履带升降 / 行驶模式"最相关的两个字段，但均为**只读反馈**。

### 5.4 [can_output.cpp](../src/cav_bywire_lvtan/src/can_output.cpp)——解码车辆反馈（CAN → 程序）

`can2pc_CB` 收到 CAN 帧后按 ID 分派到 4 个解码函数（协议见 [第 3 节表格](#3-话题与-can-帧总表)）。所有解码都用小端 `read_u16_le`，并带健壮性校验（DLC<8 丢弃、SOC>100 截断、drive_state 非法值告警、履带角越界告警）。

关键：`decode_can_0X4D2` 中

```cpp
can_output.track_angle = static_cast<double>(data[1]) - 90.0;  // 反馈履带角
if (track_angle < -50.0 || track_angle > 80.0) { /* 越界告警 */ }
```

`decode_can_0X4D1` 末尾置 `have_wheel_feedback = true`，供安全层判断是否已有轮速反馈。

### 5.5 [can_input.cpp](../src/cav_bywire_lvtan/src/can_input.cpp)（742 行）——编码命令（程序 → CAN）【重点】

这是 C++ 侧最重要的文件，决定了"到底有哪些控制量能发给车"。

#### (1) 两种 0x4C2 控制模式

文件顶部用编译期常量选择控制格式（第 36 行 `kCanControlMode`），当前为 **`DirectWheelRpm`（直接左右轮 rpm）**；另一模式 `EquivalentSpeedSteer`（等效车速+前轮转角，把差速换算成"自行车模型"下发）备用。

`wheel_rad_s_to_equivalent_command()` 展示了履带差速运动学：

```cpp
speed_mps    = 0.5 * (left_linear + right_linear);                  // 共模 → 纵向速度
yaw_rate     = (right_linear - left_linear) / (2 * LYNX_HALF_TRACK); // 差模 → 横摆
steer_rad    = atan(LYNX_L_WHEELS * yaw_rate / speed_mps);          // 等效前轮转角
```

#### (2) 控制指令回调 `control2bywire_CB`（第 341 行）

```cpp
can_input.loss_control_num = 0;  // 收到新指令，清零丢失计数

// 上层置停的四种情形之一即置停：线控关闭 / 急停 / 驻车
bool stop_requested = (msg->bywire_control_enable == 0)
                   || (msg->emerg_brake != 0)
                   || (msg->park_enable != 0);
if (stop_requested) { Stop=1; Go=0; 轮速=0; return; }

Stop=0; Go=1;
can_input.left_drive_wheel_speed_cmd  = msg->left_drive_wheel_speed_cmd;
can_input.right_drive_wheel_speed_cmd = msg->right_drive_wheel_speed_cmd;
```

> ⚠️ 注意：此回调**只取安全开关和左右轮速**，`msg->gear_cmd` 等字段被忽略。这也是为什么第 328 行的 `gear_from_standard_to_lynx()` 虽然定义了却从未被调用——它是一个**预留钩子**，将来接入履带挡位时可复用。

#### (3) 帧组装 `set_cmd_can_msg`（第 615 行）

```cpp
can_cmd_msgs.clear();
encode_4C1();       // RTK 经纬度
encode_4C2();       // 核心控制帧（Stop/Go + 轮速）
encode_4C3();       // 本地位姿
encode_4C4_4CD();   // 路径点
encode_4CE_4CF();   // 路径半径
encode_4C0();       // 剩余半径
```

`encode_4C2()` 是核心。仅当 `Stop==0 && Go!=0` 时才填入轮速（否则 B2-B7 保持 0，实现"停车即零速"）：

```cpp
data[0] = clamp(Stop, 0, 3);   // B0 停车/刹车等级
data[1] = (Go != 0);           // B1 启动标志
if (motion_enabled)
    encode_direct_wheel_rpm_payload(data, left_can_cmd, right_can_cmd); // B2-5
data[6] = 1;                    // B6 固定标志
```

其中 `wheel_rad_s_to_rpm_x10()` 完成 `rad/s → rpm → ×10 → int16` 的换算与限幅。

#### (4) 安全层 `can_cmd_safety_check`（第 628 行）【关键安全逻辑】

在每次 50Hz 下发前执行，三重保护：

1. **机械限幅**：轮速裁剪到 `±47.7 rad/s`（`kMechanicalWheelSpeedMaxRadS`），越界告警。
2. **控制超时保护**：`loss_control_num > 30`（约 0.6s 无新指令）→ 强制 `Stop=1, Go=0`。这是防止上层节点崩溃/卡死后车辆失控的关键。
3. **反馈突变限制**：正常行驶时（且已有轮速反馈），把目标轮速约束在"当前实测轮速 ± 9.4 rad/s"内，避免指令阶跃。**注意置停时会跳过此限制**（否则非零实测轮速可能把零停车目标又拉回非零）。

#### (5) 下发心跳 `publish_can_commands`（第 728 行）

```cpp
++can_input.loss_control_num;   // 每个 50Hz 周期自增；被 control 回调清零
can_cmd_safety_check();          // 安全校验
set_cmd_can_msg();               // 组装全部帧
for (frame : can_cmd_msgs) pub_can_cmd->publish(frame);  // 逐帧发布
```

### 5.6 [rtk_parse.cpp](../src/cav_bywire_lvtan/src/rtk_parse.cpp)——组装并发布 VehicleState

`rtk_CallBack` 是 `VehicleState` 的组装点，把 **RTK 定位 + 最新 CAN 反馈** 融合：

- **输入校验**：非有限值丢弃；纬度限定 30~50°（部署区）；相对上一帧跳变 >5m 丢弃（防 RTK 跳点）。
- **坐标换算**：`UTM::LLtoUTM` 把经纬度转 UTM 米（[UTM.h](../src/cav_bywire_lvtan/include/UTM.h) 提供标准换算）。
- **状态融合**：SOC/故障码→`vehicle_state`（正常/损坏/低电）；`heading=Normalise_PI(yaw)`；`yaw_rate=can_output.yaw_rate`。
- **轮速换算**：`left_drive_wheel_speed = left_rpm * 2π/60`（rpm→rad/s）。
- **挡位映射**：`driveMode2Gear(drive_state)`（0→DRIVE,1→PARK,2→REVERSE）。
- **参考点平移**：用 `move_x/move_y` 把 RTK 天线位置平移到车辆几何参考点。
- 顺带把最新位姿写回 `can_input`（供 0x4C1/0x4C3 下行使用），最后 `pub_vs->publish`。

## 6. 包三：learning_preview_controller —— Python 控制大脑

### 6.1 [controller_node.py](../src/learning_preview_controller/learning_preview_controller/controller_node.py)（798 行）——ROS 封装与控制主环【重点】

`LearningPreviewControllerNode`：

- **构造**：声明大量 ROS 参数（话题名、`control_period=0.05`、`config_file`、`enable_bywire`、`enable_plot`、`calculate_control_on_start`、`record_dir` 等）；构造控制器实例；读取固定路径原点 UTM 坐标（`origin_utm_x/y/heading`）；创建订阅 `/vehicle/vehicle_state`、发布 `/vehicle/control2bywire`、50Hz 控制定时器；可选创建实时绘图与 Excel 记录器。
- **`calculate_control_enabled`**：一个"是否真正计算控制"的开关，可由绘图界面按钮切换。关闭时仍订阅状态、刷新绘图、但只发安全停车指令——便于上车前空跑观察。

**主环 `control_loop()`（第 536 行）**逐步：

```python
self.count += 1
# ① 安全门：任一不满足 → 发安全停车
if not enable_bywire:            publish_stop("enable_bywire is false"); return
if not has_valid_reference_path: publish_stop("路径无效"); return
if self.state is None:           publish_stop("未收到车辆状态"); return

# ② 全局 UTM 位姿 → 路径局部系
x, y, psi = self.global_to_path_frame(state.x, state.y, state.heading)
r = state.yaw_rate      # 横摆角速度
u = state.speed_x       # 纵向车速

# ③ 若关闭计算：只更新绘图 + 发停车
if not calculate_control_enabled: ...; publish_stop(...); return

# ④ 组装状态并调用控制器
state_in = [x, y, psi, r, u, last_r, last_u, last_w_l, last_w_r]
out = self.controller.output(state_in)
w_l, w_r = out[12], out[13]      # 左右轮速指令

# ⑤ 填充并发布 Control
cmd.left_drive_wheel_speed_cmd  = w_l
cmd.right_drive_wheel_speed_cmd = w_r
cmd.bywire_control_enable=1; cmd.emerg_brake=0; cmd.park_enable=0
self.pub_cmd.publish(cmd)
# 记录 last_*，供下一周期的模型/学习使用
```

- **`global_to_path_frame`**：把 UTM 全局位姿平移+旋转到"路径起点为原点、起点航向为 x 轴"的局部系。
- **`make_safe_zero_cmd` / `publish_stop`**：所有异常路径的兜底，构造 `bywire_control_enable=0, emerg_brake=1, park_enable=1` 的停车指令；`publish_stop` 只在停车原因变化时打印日志（防刷屏）。
- **`compute_tracking_errors`**：不跑控制器也能算横/航向误差，供绘图。

> 🔑 **VLA 接入点**：主环第 ④⑤ 步是"决策→指令"的汇聚处。VLA 既可做成**独立节点**并行发履带指令，也可在此把 VLA 结果汇入 Control（若走加字段方案）。推荐前者以解耦重推理负载。

### 6.2 [controller.py](../src/learning_preview_controller/learning_preview_controller/controller.py)（462 行）——主类与单步 `output()`

**`__init__`**：读 YAML → 解析维度/约束/车辆/学习/权重参数 → 选控制器类型（`preview` 或 `mpc`）→ 初始化名义模型矩阵 `A/B`（调 `CalTrackWheelAu`）→ 生成参考路径 → 构建 LQR/MPC 的大权重矩阵 `Q_big/R_big`。要求 `Nu==2`（输入必为 `[W_l, W_r]`）。

**`output(StateIn)`（第 317 行）**——单步控制计算，六个阶段：

```python
# 拆解输入
x,y,psi,r,u,last_r,last_u,last_w_l,last_w_r = StateIn

# ① 扰动观测：实测状态 - 模型预测 = 未建模扰动
Obserdata = x_now - (A @ x_last + B @ u_last_vec)   # x=[u;r]
# 维护滑动样本窗（状态样本 + 扰动样本）

# ② 非参数扰动估计 g = [f_U; f_R]（Lipschitz 插值），若 useDisturbance
g = [fhatpre_scalar(u,...), fhatpre_scalar(r,...)]

# ③ 矩阵在线学习（仅高速 u>speed_threshold 时）
if useMatrixLearning and u>speed_threshold: Matrix_update_learning(...)

# ④ 找最近路径点 → 横向误差/航向误差/曲率
_, ID = FindNearestPoint(ref_path, x, y, psi, ID_last)
lateral_err, psi_err, kappa_ref = CalErr(x, y, psi, ID, ref_path)

# ⑤ 组装 3 状态误差向量，取预瞄曲率
epsi_dot = r - u_r * kappa_ref
Paper_state = [lateral_err, psi_err, epsi_dot]
kappa_preview = GetPreviewCurvature(ID, Np)

# ⑥ Preview-LQR 求解 → 左右轮速
w_l, w_r, U_seq, status = VehiclePreviewLQRControl_WithGComp(...)

# 把 20 个中间量塞进 self.out[0..19] 返回（见附录）
```

### 6.3 [preview_lqr.py](../src/learning_preview_controller/learning_preview_controller/preview_lqr.py)（508 行）——Preview-LQR 控制律【重点】

这是实际产生轮速指令的核心算法。采用 **3 状态预瞄 LQR**。

**误差模型 `CalPaperPreviewMatrix`**：状态 `x_p = [e_y, e_psi, e_psi_dot]`（横向误差、航向误差、航向误差率），控制 `W = [W_l, W_r]`：

```
x_p(k+1) = A_p x_p(k) + B_p W(k+1) + D_p·kappa(k)

        [1  Ts·Ur   0 ]        [ 0    0  ]        [      0       ]
  A_p = [0   1     Ts ]  B_p = [ 0    0  ]  D_p = [      0       ]
        [0   0     a22]        [b21  b22]        [Ur·(a22 − 1)  ]
```

其中 `a22, b21, b22` 来自在线学习得到的 U-R 模型 `A/B`。可选地（`use_df_R_dr_in_lqr_A`）把学到的横摆残差斜率 `df_R/dr` 注入 `A_p[2,2]`，进一步补偿非线性。

**增益求解 `CalPreviewLQRGain`**：解离散代数 Riccati 方程 `solve_discrete_are(A_p,B_p,Q,R)` 得 `P`，再算：

- **反馈增益** `K_b = (R + Bᵀ P B)⁻¹ Bᵀ P A`
- **预瞄前馈增益** `K_f`（N+1 项，沿闭环转置矩阵 `(A_p−B_p K_b)ᵀ` 递推）——利用未来 N 步的曲率提前打舵。

**总控制律 `VehiclePreviewLQRControl_WithGComp`**：

```python
W_base = solve_base_wheel_speed_UR(kappa_now, g)  # ① 维持 U_r 车速的基础轮速
W_c    = -K_b @ x_p - K_f @ C_R                    # ② 反馈 + 预瞄前馈修正
W_cmd  = W_base + W_c                               # (g 已在 W_base 中考虑)
# ③ 增量限幅：|W_cmd - w_last| ≤ Delta_InputMax
# ④ 绝对限幅：lb_temp ≤ W_cmd ≤ ub_temp
w_l, w_r = W_cmd
```

- **`solve_base_wheel_speed_UR`**：求稳态基础轮速——期望 `U=u_r, R=kappa·u_r`，由 `X_ref = A·X_ref + B·W_base` 反解 `W_base`。前 `g_warmup_steps` 步不用学到的 `g`（预热）。
- **`CalGCompensation` / `_YawOnly`**：把扰动 `g` 用 `W_g = −pinv(B)·g` 抵消（全通道或仅横摆通道）。
- **`GetPreviewCurvature`**：按 `预瞄步长 ≈ u_r·T / 平均点间距` 沿参考路径取未来 N+1 个曲率。
- **异常兜底**：Riccati 求解失败时复用上一次有效增益，再不行则维持上一轮速。

### 6.4 [learning.py](../src/learning_preview_controller/learning_preview_controller/learning.py)（423 行）——在线学习【重点】

在线辨识履带车的 U-R 模型 `x(k+1) = A·x(k) + B·W(k+1) + g`，其中 `x=[u; r]`。

**多种学习模式**（由 config 开关组合，见 `Matrix_update_learning`）：

- `update_Au / update_Ar`：单参数辨识（只更新 `A` 对角元）。
- `update_ABu / update_ABr`：分通道辨识 `A` 行 + `B` 行（u 通道学第 1 行，r 通道学第 2 行）。
- `update_AB_full`：一次性辨识整个 2×2 的 `A` 和 `B`（需激励充分，`rank(Xminus)≥4` 才更新）。

**核心机制**：

- **滑动窗最小二乘**：`push_window` 维护数据窗 `Xplus/Xminus`，参数估计 `θ = Xplus @ pinv(Xminus)`。
- **触发式更新 `should_update_param`**：由 `usingTrigger` 控制。用 `calculate_IQII_hat`（基于 Gram 行列式的信息增量判据）判断新数据是否带来足够激励（`IQII > epsilon`），避免在无激励时污染估计。
- **非参数扰动估计 `fhatpre_scalar`**：核心是 Lipschitz 上下包络插值——对每个历史样本算 `上界=g_j+L·|x−x_j|+e`、`下界=g_j−L·|x−x_j|−e`，取 `f = 0.5·(min上界 + max下界)`。这是 KI/LACKI（Kinky Inference / Lazily Adapted Constant Kinky Inference）类方法，给出对未建模非线性 `f(x)` 的鲁棒估计。`fhatpre/bhatpre` 是其向量版本。

### 6.5 [vehicle_model.py](../src/learning_preview_controller/learning_preview_controller/vehicle_model.py)——履带差速动力学

由物理参数（质量 `m`、半履带距 `b`、转动惯量 `Iz`、轮半径、采样 `T`）与当前 `Au_est` 计算名义 `A/B`：

```python
ku = m*(1 - Au_est)/(2T);  wkomega = ku * wheel_radius
A = [[Au_est, 0],
     [0,      1 + T*(-b²·ku)/(2Iz)]]
B = [[ T·wkomega/m,          T·wkomega/m       ],   # 共模 → 纵向
     [-T·b·wkomega/(2Iz),    T·b·wkomega/(2Iz) ]]   # 差模 → 横摆
```

`B` 的两行清楚体现"左右轮共模驱动纵向、差模驱动横摆"的差速本质。`CalTrackWheelAr` 单独更新 `A[1,1]=Ar_est`。

### 6.6 [path.py](../src/learning_preview_controller/learning_preview_controller/path.py)（460 行）——参考路径生成与误差计算

- **三种路径生成器**（由 `path.type` 选择）：
  - `GenerateStraightCircleRef`：直线 + 圆弧 + 直线。
  - `GenerateSineRef`：直线 + 正弦 + 直线。
  - `GenerateDoubleLaneChangeRef`：双移线（用 tanh 叠加，标准换道测试轨迹）。
  三者都**按弧长定长采样**（`ds = v_ref·Ts`），返回 `[x, y, heading, curvature]` 四列。
- `NormalizeRefPathStart`：把生成路径平移旋转到起点为 `(0,0,0)`。
- `FindNearestPoint`：在上次索引附近的窗口内找最近点（带航向一致性 `|Δψ|<π/2` 过滤），高效且防跳点。
- `CalErr`：投影法算横向误差 `e_y = −sin(ψ_r)·dx + cos(ψ_r)·dy` 与航向偏差 `Δψ`。

### 6.7 [compatibility.py](../src/learning_preview_controller/learning_preview_controller/compatibility.py)（240 行）——QP-MPC 备选控制器

当 `controller.type: mpc` 时启用（当前 `preview_params.yaml` 用 preview，故未激活；`mpc_params.yaml` 用 mpc）。

- **MPC 状态** `[lateral_error, heading_error, speed, yaw_rate]`（4 维），输入 `[W_l, W_r]`。
- `CalMPCMatrix`：构造 4×4 预测模型（含学习到的 `A/B` 与扰动 `d`）。
- `VehicleQPControl`：用 cvxpy/OSQP 求解带约束 QP——构造预测矩阵 `Abar/Bbar/G`，代价 `‖X−X_ref‖²_Q + ‖ΔU‖²_R`，约束含增量限幅 `Delta_InputMax` 与绝对限幅 `lb/ub`。求解失败时维持上一轮速。

## 7. 一个 20ms 控制周期里发生了什么

把两侧串起来，完整走一遍闭环：

1. **车辆** 通过 CAN 上报 `0x4D1~0x4D4`。**C++ 驱动** `can2pc_CB` 解码进 `can_output`。
2. **RTK** 发来 `/car/gps`。**C++ 驱动** `rtk_CallBack` 融合定位 + `can_output`，发布 `/vehicle/vehicle_state`。
3. **Python 控制器** 50Hz 定时器触发 `control_loop`：读最新 `VehicleState` → 转局部系 → `controller.output()` 跑 [扰动观测 → 学习 → 找最近点 → Preview-LQR] → 得 `w_l, w_r` → 发布 `/vehicle/control2bywire`。
4. **C++ 驱动** `control2bywire_CB` 收到，更新 `can_input` 的 Stop/Go 与轮速，并清零 `loss_control_num`。
5. **C++ 驱动** 50Hz 定时器 `publish_can_commands`：`can_cmd_safety_check`（机械限幅 / 超时保护 / 突变限制）→ `set_cmd_can_msg` 编码 `0x4C1~0x4CF` → 发到 `/to_can_bus` → **车辆执行**。

任一环节异常（无路径、无状态、控制超时 0.6s）都会退化为**安全停车**。

### 术语

- **U-R 模型**：以纵向速度 `u` 与横摆角速度 `r` 为状态的 2 维车辆模型 `x=[u; r]`。
- **差速驱动**：左右履带共模 → 纵向速度，差模 → 横摆，本车无独立转向机构。
- **Preview-LQR**：带未来曲率预瞄前馈的线性二次型调节器。
- **触发式学习（IQII）**：仅当新数据激励足够（信息增量超阈）时才更新模型参数。
- **KI/LACKI 扰动估计**：基于 Lipschitz 上下包络插值的非参数函数估计。
- **`track_angle`**：车辆上报的履带角（度，[-50,80]），**当前仅反馈、无命令通道**。

## 10. 面向 VLA 接入的要点小结

1. **打滑率输入**：所需数据（`left/right_drive_wheel_speed`、`speed_x`）已在 `VehicleState` 中，VLA 节点订阅 `/vehicle/vehicle_state` 即可自算，无需改驱动。
2. **视觉输入**：当前系统完全没有相机话题，需全新引入（`sensor_msgs/Image` 等），与现有代码正交、风险低。
3. **履带升降输出（最大缺口）**：当前**只有 `track_angle` 只读反馈，无任何下行命令帧**。要让 VLA 输出作动，需要打通整条链路：
   - 向车厂索取"命令履带升降"的 CAN 协议（ID/字节/量纲）；
   - 在 `cav_msgs` 新增履带指令消息（或给 `Control.msg` 加字段）；
   - 在 C++ 驱动新增订阅回调 + `encode_track()`（可复用闲置的 `gear_from_standard_to_lynx` 钩子与 `0x4C2` 的空闲 B7）+ 安全校验；
